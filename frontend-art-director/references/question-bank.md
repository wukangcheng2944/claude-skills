# Question Bank

Use these only when the answer would materially change the critique or direction. Ask the smallest set of questions needed.

## Core default questions

1. Who is the target audience?
2. What should this page or screen make the user do, understand, or feel?
3. What kind of page is this: landing page, product page, dashboard, internal tool, onboarding, settings, or something else?
4. What tone should it communicate: premium, playful, calm, technical, editorial, bold, minimal, etc.?
5. Are there references to emulate or avoid?
6. Are there hard constraints: mobile first, existing brand colors, framework, deadline, dense data, accessibility, or engineering limits?

## When the user provides a screenshot

Ask only if not obvious:
- Is this already live, or just a draft?
- What exactly feels wrong to you: too generic, too busy, too empty, too cheap, too hard to scan, or something else?
- Do you want a critique only, or a redesign direction too?

## When the user provides a link

- Which page or section should be reviewed first?
- Should the review focus more on visual direction, layout, or conversion/storytelling?

## When the user provides code

- Should the feedback stay at the visual/system level, or do you also want implementation-level suggestions?
- Is the current UI roughly correct but visually weak, or do you want the structure reconsidered too?

## When the user provides only a brief

Prioritize:
- audience
- page goal
- page type
- desired tone
- reference style

## When the user wants generation after critique

Clarify:
- do you want a critique, a creative brief, a prompt, code, or all of them?
- should the result preserve the current structure or allow a more radical redesign?
